# Repository Guidelines

## Scope

This repository exists only to package portable Claude Code CLI artifacts with GitHub Actions. Do not commit generated binaries, archives, or downloaded npm dependencies (`.work/`, `dist/`, `node_modules/`, `*.tar.gz`, `*.zip`, `*.sha256`).

## Current state

- This repository builds portable Claude Code bundles for eight GitHub-hosted runner targets:
  - `ubuntu-22.04` / `linux-x64` (glibc)
  - `ubuntu-22.04-arm` / `linux-arm64` (glibc)
  - `ubuntu-22.04` / `linux-x64-musl`
  - `ubuntu-22.04-arm` / `linux-arm64-musl`
  - `macos-15` / `darwin-arm64`
  - `macos-15-intel` / `darwin-x64`
  - `windows-2022` / `win32-x64`
  - `windows-11-arm` / `win32-arm64`
- The pipeline has three stages:
  1. **build** — fetch the platform-specific npm package via `npm pack`, extract the binary, wrap it with a launcher script, and archive (`.tar.gz` for Linux/macOS, `.zip` for Windows).
  2. **verify** — download each artifact and smoke-test it in a clean environment: native extraction on the matching runner for glibc Linux, macOS, and Windows; Alpine Docker for musl. Both `--version` and a real API round-trip (`--model haiku`) are checked.
  3. **publish** — only runs on `release.published`. Uploads assets to the release and regenerates the README files from templates.
- Release notes are generated by `claude` itself (via `--allowedTools WebFetch`), which fetches the upstream `anthropics/claude-code` changelog and summarizes it alongside the portable packaging info.
- Release documentation is intentionally pinned to the real published release, not resolved dynamically at read time.

## Source of truth

- Use the official `@anthropic-ai/claude-code` npm package as the upstream source, via its per-platform sub-packages (e.g. `@anthropic-ai/claude-code-linux-x64-musl`).
- Always use `npm pack` rather than `npm install` to fetch the platform-specific tarball — `npm install` rejects mismatched `os` / `libc` with `EBADPLATFORM`, blocking cross-platform musl builds on glibc runners.
- The launcher must inject `DISABLE_AUTOUPDATER=1` so the portable binary never attempts self-update on an air-gapped host.
- Preserve upstream version information in `portable-metadata.json`.

## README and template rules

- `README.md` and `README_zh.md` are **generated files**. Do not manually edit them under any circumstances.
- The source of truth for README content is:
  - `templates/README.md.j2`
  - `templates/README_zh.md.j2`
  - `scripts/render-readme.py`
- Any change to wording, structure, examples, platform lists, or release-pinned documentation behavior **must** be made in the templates or the renderer, never in the rendered outputs. The workflow will overwrite any manual README edits on the next release.
- The English and Chinese READMEs must cross-link to each other at the top of the document. Keep the two templates in sync in structure and section order.
- Keep the generated README files pinned to the currently published release and actual uploaded assets. Do not reintroduce `latest` download links into the published README outputs.

## Editing rules

- Keep documentation, workflow files, and helper scripts in English. User-facing docs may be bilingual (EN + ZH) but the source must stay in English for maintainability.
- Prefer small, reviewable changes.
- Do not hardcode secrets, tokens, or user-specific paths.
- Validate workflow behavior with `workflow_dispatch` after any meaningful CI change. The build job is safe to re-run: it does not publish or mutate the release unless the event is `release.published`.
- When changing README behavior, edit the templates and regenerate locally with `python3 scripts/render-readme.py` against a real release JSON before committing.

## Artifact expectations

- Bundles must stay self-contained and portable — everything needed to run Claude Code ships inside the archive.
- Bundles must not contain user state from `~/.claude/` or any host-specific configuration.
- The launcher (`claude` / `claude.cmd` / `claude.ps1`) must resolve symlinks correctly so users can `ln -sf` it into `~/.local/bin/`.
- Linux and macOS artifacts ship as `.tar.gz` only. Windows artifacts ship as `.zip` only. Each archive is paired with a `.sha256` file. No duplicate format uploads.

## Secrets

The workflows require two repository secrets:

- `ANTHROPIC_API_KEY` — used by the `verify` smoke tests and by the release-notes generator (Claude calls itself).
- `RELEASE_WORKFLOW_TOKEN` — a PAT with `repo` scope on this repository. Used by `gh release create/edit/upload` inside the workflows and by `actions/checkout` in the publish job so it can push the regenerated README back to `main`.

Do not commit these or any other secret.
